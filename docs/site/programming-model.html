Programming Model

A FuncDraw project is just a collection of pure expressions that describe what to render and how to frame it. Understanding how the player breaks those expressions into main, view, collections, and modules makes it easy to reuse components or render the same model from the CLI.

Model anatomy

  • Main expression – the entry point which when evaluated emits the list (or lists) of primitives you want on screen. The player evaluates this tab every time you edit.
  • View expression – an expression that defines the extent of the viewable coordinates in the form { minX, minY, maxX, maxY }, giving the renderer deterministic camera bounds.
  • Collections – gather expressions, other collections, or modules into a single namespace. Everything inside can be referenced with dot syntax (collections.hero, collections.metrics.cards, etc.), making it easy to wire helpers together.
  • Modules – act like callable helpers. Each module exposes a single expression named eval; referencing the module runs eval, which can use any private expressions in that folder before yielding a value.

The examples below come from docs/examples/programming-model/. Each PNG was rendered with:

fd-cli --root docs/examples/programming-model --view view --expression <name> --format png --out docs/assets/programming-model/<name>.png

Simple main + view example

A minimal pair of expressions is all you need to render a scene. The view tab fixes the camera, while main composes a few reusable helpers. The JavaScript equivalents live next to these files in docs/examples/programming-model/ if you prefer writing return statements.

View (FuncScript)

{
  eval {
    minX: -40;
    minY: -28;
    maxX: 40;
    maxY: 32;
  };
}

Main (FuncScript)

{
  palette: modules.palette;
  canvas:{
    type:'rect';
    data:{
      position:[-38,-26];
      size:[76,52];
      fill: palette.background;
      stroke: palette.accent;
      width:0.6;
    };
  };
  grid:{
    type:'rect';
    data:{
      position:[-36,-24];
      size:[72,48];
      fill:'transparent';
      stroke: palette.muted;
      width:0.3;
    };
  };
  eval [canvas, grid, collections.hero, collections.metrics];
}

Rendered with fd-cli --expression main --view view

Need a richer composition? The packaged project under docs/examples/programming-model/ layers additional helpers (hero banners, metric cards, etc.) that you can inspect or run as-is.

Collections

Collections turn folders into reusable namespaces for expressions, other collections, or modules. In this sample, the collections folder automatically exposes hero (the banner) and metrics (the row of cards) simply because tabs with those names live inside the folder—no extra aggregator needed. Importing a .fdmodel archive merges those folders into your workspace so you can reuse helpers without copy/pasting tabs by hand. For deeper hierarchies (nested folders, multiple previews), explore the packaged project under docs/examples/programming-model/collections/.

Modules

Modules gather constants—palettes, spacing, reusable helper shapes—behind a callable folder. Each module exposes one expression named eval, and referencing the module (modules.palette) simply runs eval. Any additional expressions in the module stay private to that folder, so outside callers can’t reach them directly; eval wires everything together on their behalf. Inside the folder, those private expressions are still available by name, so eval can build complex results without leaking the intermediate pieces. The packaged module examples under docs/examples/programming-model/modules/ add more layers if you want to inspect a multi-file setup.

Rendered with fd-cli --expression modules_preview
// modules/palette/eval.fs
{
  shades: shades;
  accents:{
    primary:'#38bdf8';
    secondary:'#facc15';
  };
  eval {
    background: shades.background;
    surface: shades.surface;
    accent: accents.primary;
    muted: shades.muted;
    highlight: accents.secondary;
    text: '#f8fafc';
  };
}
